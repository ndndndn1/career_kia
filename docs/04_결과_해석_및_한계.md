# 04. 결과 해석 및 한계

> 학습·분석이 완료된 시점의 실제 수치 요약. 재현은 `bash scripts/reproduce.sh`.

## 4.1 모델 성능 요약

`GroupKFold(operator_id)` 5-fold 교차검증 (희소 양성 3.4%).

| 모델 | ROC-AUC | PR-AUC | F1 | P@R=0.9 |
|---|---|---|---|---|
| Logistic Regression | 0.934 | 0.528 | 0.395 | 0.136 |
| Random Forest | 0.960 | 0.680 | 0.621 | 0.270 |
| **Hybrid LGBM + pyGAM** | **0.962** | **0.857** | **0.776** | **0.611** |

- 하이브리드 모델이 PR-AUC 에서 단일 모델 대비 +0.18 개선 — 희소 고장 감지의 실용성 확보
- Recall 0.9 유지 시 Precision 0.61 → 실제 현장 운영 기준으로 오탐을 허용 수준에서 통제

## 4.2 주요 XAI 인사이트

전역 SHAP 상위 기여 변수 (발췌):

1. **`t_rms_mean`** — 진동 RMS 평균 (에너지 지표)
2. **`f_env_BPFO_max`** — 외륜 결함 특성 주파수 진폭
3. **`Torque [Nm]`** — 공정 토크
4. **`Tool wear [min]`** — 공구 마모 시간
5. **`Process temperature [K]`** — 공정 온도

상위 상호작용 쌍:
- 대기 온도 ↔ 회전 속도 (강도 0.50)
- 대기 온도 ↔ 공정 온도 (강도 0.33)
- 회전 속도 ↔ 토크 (강도 0.26)

고위험 배치 10건에 대한 한국어 자연어 설명이 `xai_artifacts/top_risk_narratives.txt` 에 저장됨. 대시보드 2페이지("불량원인 설명") 에서 배치 단위로 조회 가능.

## 4.3 인과 분석 결과 (ATE)

DAG 기반 backdoor 조정 + linear regression estimator.

| 처치 변수 | 기준 → 개입 | ATE (고장 확률 변화) | Refutation (random_common_cause) | Refutation (data_subset) |
|---|---|---|---|---|
| Tool wear | 50 → 200 분 | **+4.47 %p** | +4.47 %p | +4.46 %p |
| Torque | 40 → 60 Nm | **+6.76 %p** | +6.76 %p | +6.85 %p |
| Rotational speed | 1600 → 1300 rpm | **+5.91 %p** | +5.91 %p | +5.91 %p |

- 모든 ATE 가 두 종류의 refutation 에서 원 추정치와 거의 일치 → 교란변수 통제 후 효과 안정적
- 회전 속도 저속에서 고장 확률이 올라가는 결과는 도메인 관점에서 방열 불량·체류 시간 증가로 해석 가능

### 현장 제언 (예시)
- 공구 마모 임계를 기존 운영 가정 200분에서 150분으로 앞당기면 약 1.5 %p 고장 감소 예상
- 토크를 60Nm → 40Nm 감소 시 약 6.8 %p 고장 감소 — 대신 사이클 타임 영향 분석 병행 필요
- 저속 구간(1300rpm 이하)에서 공정 시간 감시 강화 권장

## 4.4 한계 및 향후 과제

### 데이터 측면
- 공개 데이터셋 조합이라 실제 자동차 파워트레인 공정과 완전 일치하지 않음
- CWRU 진동은 제한된 결함 유형만 포함 — 현장 교체 시 다양한 결함 유형 확장 필요
- 합성 `batch_id` 매핑에 도메인 가정이 있음 (JD의 "false negative / early warning" 시나리오)

### 모델 측면
- `precision_at_recall_0.9` 가 0.61 → 운영 환경에서 오탐 허용 기준은 라인별 재설정 필요
- pyGAM 메타 모델은 현 피처 6개에 한정 — 도메인 전문가 리뷰 후 확장 가능

### 인과 측면
- 현재 추정기는 `backdoor.linear_regression` 중심 — 실제로는 이진 결과이므로 logistic 또는 DML 적용이 이론적으로 더 타당
- `placebo_treatment_refuter` 는 DoWhy 0.14 호환 이슈로 제외 — 버전 업 시 재활성화
- 시계열 인과(PCMCI) 는 개념적 소개만 제공; 현장 데이터는 tigramite 로 본격 적용 권장

### 배포 측면
- MLflow 는 파일 백엔드 (데모). 운영 시에는 DB 백엔드(Postgres)로 전환 권장
- 드리프트 감지는 PSI + KS 기본형. 실운영에서는 Evidently 같은 프레임워크로 알람/리포트 자동화
- 실시간 스트리밍(Kafka 등) 은 본 프로젝트 범위 밖 — 배치 시뮬레이션으로 대체
